## 重點摘要

Tim 分享了自己曾犯過的五個技術面試錯誤，並解釋如何避免。他成功拿到了 Microsoft、Shopify、Google 的 Offer，核心心得是停止這五個錯誤比刷更多題更重要。

## 五大錯誤詳解

### 錯誤 1：隨機刷 LeetCode
- **問題**：沒有計劃地隨機刷題，解了 200 題卻在面試中腦袋一片空白
- **根本原因**：記住的是答案而非模式（memorizing answers vs learning patterns）
- **後果**：題目稍有變化就完全不知所措，綠色勾勾給你虛假的進步感

### 錯誤 2：被動學習
- **問題**：看 YouTube、讀文章、瀏覽教程，感覺學了很多但實際什麼都沒做
- **比喻**：就像看別人做伏地挺身以為自己在變強
- **後果**：面試時認得出問題、覺得看過，但就是解不出來
- **關鍵**：被動學習只帶來「熟悉感」（familiarity），不是「信心」（confidence）

### 錯誤 3：忽略溝通能力
- **問題**：以為技術面試只看代碼，但面試官同時在評估思考過程、提問能力、解釋推理、處理卡住的方式
- **紅旗行為**：
  - 沉默五分鐘不說話
  - 不先說明思路就開始寫程式
  - 能解題但無法解釋為什麼可行
- **根因**：大多數人獨自練習，從不練習邊想邊說
- **Tim 觀察**：見過技術很強但因溝通問題持續失敗的工程師

### 錯誤 4：基礎薄弱
- **問題**：急著跳到 Medium/Hard 題，但實際上不理解基礎
- **表現**：
  - 無法解釋 Hash Map 底層原理
  - 分不清 BFS 和 DFS（不 Google 的話）
  - Big O 分析不穩
- **致命時刻**：面試官追問 "為什麼用 Hash Map 而不是 Array？" → 答不出來
- **比喻**：不能在脆弱的地基上蓋房子

### 錯誤 5：沒有學習結構
- **問題**：隨機看 DP 影片 → 做 Array 題 → 讀 System Design 文章 → 買書看幾章就放著
- **後果**：花六個月準備仍然失敗，500 小時分散且無聚焦
- **缺少的**：課程體系、學習順序、進度追蹤、弱點識別

## 解決建議

1. **學模式而非記答案**：理解底層概念，建立直覺
2. **主動練習**：自己動手寫，不只是看別人寫
3. **練習邊說邊想**：模擬面試情境，練習口頭表達思路
4. **先打好基礎**：資料結構、演算法、時間複雜度分析
5. **建立結構化學習計劃**：按類別和難度有序推進
